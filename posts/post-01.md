코드 리뷰는 기술적인 과정이기도 하지만 사람 사이의 소통이기도 하다. 잘 하는 것과 못 하는 것의 차이는 생각보다 크다.

## 리뷰 요청할 때 (Author)

### PR을 작게 유지하라

리뷰어가 한 번에 검토하기 좋은 크기는 **200~400줄** 정도다. 1000줄짜리 PR은 제대로 리뷰받기 어렵다.

### 컨텍스트를 제공하라

PR 설명에 다음을 포함하면 리뷰가 훨씬 쉬워진다.

```markdown
## 변경 내용
- 사용자 인증 로직을 JWT 기반으로 전환

## 이유
- 세션 기반 방식의 서버 메모리 문제 해결
- 수평 확장 대비

## 테스트 방법
1. /login으로 POST 요청
2. 반환된 토큰으로 /api/me 요청
```

### 자기 리뷰 먼저

PR 올리기 전에 diff를 직접 한 번 읽어보자. 본인이 발견할 수 있는 문제를 리뷰어에게 떠넘기지 않는 것이 예의다.

## 리뷰할 때 (Reviewer)

### 명령이 아닌 제안으로

```
# 나쁜 예
이 함수를 분리하세요.

# 좋은 예
이 함수가 여러 책임을 갖고 있는 것 같아서요, 분리하면 테스트하기도 편할 것 같은데 어떻게 생각하세요?
```

### 칭찬도 남겨라

좋은 코드를 발견했을 때 칭찬 한 마디는 팀 분위기를 만든다.

### 중요도를 표시하라

모든 코멘트가 동등하지 않다. 레이블을 붙이면 우선순위가 명확해진다.

- `[필수]` 반드시 수정해야 함
- `[제안]` 고려해볼 만한 개선
- `[질문]` 이해를 위한 질문
- `[nit]` 사소한 스타일 지적

## 리뷰 받았을 때

- 방어적으로 받아들이지 말자. 코드에 대한 피드백이지 나에 대한 공격이 아니다.
- 동의하지 않으면 이유를 들어 대화하자.
- 반영했으면 코멘트에 완료 표시를 남기자.

코드 리뷰 문화가 좋은 팀은 코드 품질도 자연스럽게 높아진다. 기술적 역량만큼 소통 방식도 중요하다.
